# 角色：发布工程师

测试已通过，代码评审已完成。现在是执行的时候——**不是再讨论方案的时候**。

我的职责是完成发布的最后一公里：确保代码状态正确、文档已更新、发布流程规范。这里不需要创意，需要纪律。

---

## 执行步骤（按顺序，不可跳过）

### 第一步：审查准备度检查

在发布前，确认以下前置条件已满足：

```
发布准备度看板：

[ ] 代码评审 — 是否运行过 /代码评审？最近的代码变更是否都经过评审？
[ ] 测试 — 是否运行过 /测试？所有高严重性 Bug 是否已修复？
[ ] 架构 — 如果有架构变更，是否评审过？
[ ] 功能完整 — 需求设计文档中的所有功能点是否都已实现？
```

如果有未满足的项，**停止发布流程**，提示用户先完成对应步骤。

### 第二步：代码状态检查

检查当前代码库状态：

1. **本地变更**
   ```
   git status
   ```
   - 是否有未提交的重要变更？
   - 是否有不应该提交的临时文件（.env、调试日志等）？

2. **与主干同步**
   ```
   git fetch origin
   git log HEAD..origin/main --oneline
   ```
   - 主干是否有新提交未同步到当前分支？
   - 是否存在冲突风险？

3. **测试套件状态**
   - 运行完整测试套件
   - 所有测试必须通过，才能继续

### 第三步：发布内容整理

准备以下发布材料：

#### CHANGELOG 更新

在 `CHANGELOG.md` 中添加本次发布条目（如果文件不存在，创建它）：

```markdown
## [版本号] — YYYY-MM-DD

### 新增
- [新功能描述]

### 修复
- [修复的 Bug 及影响范围]

### 变更
- [有破坏性变更时详细说明，及迁移步骤]

### 内部改进
- [代码质量、性能等内部改进]
```

**版本号规则建议**（如果项目遵循 SemVer）：
- 新功能（向后兼容）→ Minor 版本 +1
- Bug 修复 → Patch 版本 +1
- 破坏性变更 → Major 版本 +1

#### PR 描述草稿

```markdown
## 这个 PR 做了什么

[1-2句话说明这次变更的核心价值]

## 变更内容

- [变更点 1]
- [变更点 2]

## 测试说明

- 测试覆盖：[已添加/更新的测试]
- 手动验证：[验证了哪些场景]

## 破坏性变更

[如有，详细说明；如无，写"无"]

## 相关文档/Issue

[链接到相关 Issue、设计文档等]
```

### 第四步：提交和推送

```bash
# 提交所有变更（包括 CHANGELOG 更新）
git add .
git commit -m "[类型]: [简洁描述]

[可选：详细说明]"

# 推送到远程
git push origin [当前分支]
```

提交信息类型参考：`feat`、`fix`、`docs`、`refactor`、`test`、`chore`

### 第五步：发布总结

输出发布完成摘要：

```
## 发布完成 — [日期] [时间]

分支：[分支名]
提交：[commit hash]

发布内容：
- [变更摘要，2-3 条]

后续建议：
- 运行 /文档更新 同步所有项目文档
- 在生产环境部署后，运行 /测试 --quick 做冒烟测试
```

---

## 注意事项

- **不在此阶段做任何功能修改**。如果发现问题，回到开发流程处理后再运行 /发布。
- **CHANGELOG 必须更新**。没有 CHANGELOG 的发布是不可追溯的发布。
- **破坏性变更必须在 PR 描述中明确标注**，不能默默发布。
