# Skills 使用手册

借鉴 [gstack](https://github.com/garrytan/gstack) 设计理念，为 Windows + Antigravity 环境打造的 AI 工程团队工作流。

> **核心理念**：每个 `/命令` 不只是提示词，而是一个专家角色的完整**思维模式切换**。给你的 AI 一个清晰的角色，而不是一个模糊的请求，输出质量会有质的差别。

---

## 完整的 Sprint 工作流

```
/需求梳理  →  /ceo评审  →  /架构评审  →  [写代码]  →  /代码评审  →  /测试  →  /发布  →  /文档更新
```

**每个阶段都有文档产出，会自动传递给下一个阶段。**

需要时随时使用：
- `/调试` — 当你不知道 Bug 在哪时
- `/设计评审` — 当需要 UI/UX 评审时  
- `/复盘` — 每周末或里程碑节点

## Skills 总览

| 命令 | 角色 | 触发时机 | 产出文件 |
|------|------|---------|---------|
| `/需求梳理` | YC 产品合伙人 | **任何新功能开始前，第一步必跑** | `文档/需求设计文档.md` |
| `/ceo评审` | Brian Chesky 创始人模式 | 需求梳理完成后 | 追加到需求设计文档 |
| `/架构评审` | 技术主管/工程经理 | CEO 评审通过后 | `文档/架构设计文档.md` |
| `/代码评审` | 偏执 Staff Engineer | 代码提交前/功能完成后 | 直接修复 + 输出评审总结 |
| `/调试` | 系统级调试专家 | 不知道 Bug 在哪时 | 根因分析报告 |
| `/测试` | QA 主管 | 功能实现完毕后 | `任务和测试过程/测试报告.md` |
| `/发布` | 发布工程师 | 测试通过后 | CHANGELOG 更新 + PR 描述 |
| `/文档更新` | 技术写作工程师 | 发布后 | 更新所有漂移文档 |
| `/复盘` | 数据驱动工程经理 | 每周末/里程碑节点 | `AI沟通记录/周复盘.md` |
| `/设计评审` | 会写代码的资深设计师 | UI/UX 变更后 | 设计评审报告 + 直接修复代码 |

## 快速上手（10 分钟）

**第一次使用，按这个顺序走：**

1. 告诉我你想做什么（任何新功能/项目想法）
2. 说 `/需求梳理` — AI 会追问你，别急着跳过
3. 通过后说 `/ceo评审` — 寻找 10 星体验
4. 通过后说 `/架构评审` — 锁定技术方案再写代码
5. 写完代码后说 `/代码评审` — 发现隐藏 Bug
6. 说 `/测试` — 在用户发现之前找到 Bug
7. 说 `/发布` — 规范地交付

**花 5 分钟在 `/需求梳理` 上，能省下 5 小时的返工。**

## 设计原则（继承自 gstack）

1. **角色专注** — 每个 Skill 只做一件事，切换角色时切换整个思维模式
2. **文档驱动** — 每个阶段都有产出文件，自动传递给下一个阶段
3. **偏执审查** — `/代码评审` 假设 CI 可以过但代码仍可能炸生产
4. **最小改动** — 修复 Bug 只动必要代码，绝不引入无关 diff
5. **铁律零猜测** — `/调试` 禁止在找到根因前修改任何代码

## 与 gstack 的关系

本框架借鉴了 gstack 的设计哲学，但做了以下适配：

| 方面 | gstack | 本框架 |
|------|--------|-------|
| 运行环境 | macOS / Linux + Claude Code | Windows + Antigravity |
| 触发方式 | `~/.claude/skills/` 斜杠命令 | `.agents/workflows/` 中文指令 |
| 运行时依赖 | Bun + Playwright | 无需额外安装 |
| 语言 | 英文 | 中文 |
| 浏览器能力 | `/browse` 命令 | browser_subagent 工具 |
